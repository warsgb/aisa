# WPS 365 技术栈侦探技能

从客户 JD（职位描述）反推技术架构，识别技术债务信号，分析 WPS 365 部署兼容性，生成技术对接话术。

## 触发条件

当用户提供以下任一信息时，启动本技能：

1. **JD 文件**：客户发布的招聘 JD（上传文件）
2. **JD 文本**：直接粘贴的 JD 文本内容
3. **公司名称**：仅提供公司名称，自动搜索 JD
4. **招聘链接**：客户官网或招聘网站的 JD 链接

**示例**：
- "帮我分析这个JD：[JD文本]"
- "从招商银行的JD推断技术栈"
- "分析这个招聘链接：[链接]"

## 使用流程

### 第一步：获取 JD 内容

#### 优先级1：用户上传文件（最优先）

如果用户上传了 JD 文件：
- 使用 Read 工具读取文件内容
- 提取关键信息：技术栈、工具、平台、框架

#### 优先级2：用户提供 JD 文本

如果用户直接粘贴了 JD 文本：
- 直接使用文本内容进行分析

#### 优先级3：JD 链接

如果用户提供了 JD 链接：
- 使用 WebReader 工具获取 JD 内容
- 支持的来源：公司官网、招聘网站（智联/前程无忧/Boss直聘）

#### 优先级4：仅公司名称（自动搜索）

**数据源扩展（按优先级）**：

**P1: 客户官网招聘页面**（最准确）
- 搜索关键词：`公司官网 + 招聘`
- 搜索关键词：`公司官网 + 人才招聘`
- 使用 WebReader 获取 JD

**P2: 招聘网站**（覆盖面广）
- 智联招聘：https://www.zhaopin.com/
- 前程无忧：https://www.51job.com/
- Boss直聘：https://www.zhipin.com/
- 搜索关键词：`公司名称 + 技术岗位`

**P3: LinkedIn/脉脉**（技术人员分享）
- LinkedIn：搜索公司 + 技术人员
- 脉脉：搜索公司 + 技术团队

**P4: GitHub/Gitee**（开源仓库）
- GitHub：搜索公司名称
- Gitee：搜索公司名称
- 了解技术栈通过开源项目

### 第二步：技术栈识别

从 JD 中提取技术栈信息，按以下维度分类：

#### 1. 编程语言

**识别关键词**：
- **Java**：Java, Spring, Spring Boot, Spring Cloud, MyBatis
- **.NET**：.NET, C#, ASP.NET, .NET Core
- **Python**：Python, Django, Flask, FastAPI
- **JavaScript/前端**：JavaScript, TypeScript, React, Vue, Angular
- **Go**：Go, Golang
- **其他**：PHP, Ruby, C/C++, Rust

**技术特征推断**：
- Java + .NET → 传统企业级架构，可能存在异构系统
- Java + Spring Cloud → 微服务架构
- .NET + AD → 微软技术栈
- Go + Kubernetes → 云原生架构

#### 2. 数据库

**识别关键词**：
- **关系型**：MySQL, PostgreSQL, Oracle, SQL Server, DM（达梦）, Kingbase（人大金仓）
- **NoSQL**：MongoDB, Redis, Elasticsearch
- **国产数据库**：达梦、人大金仓、南大通用、OceanBase

**技术特征推断**：
- Oracle/SQL Server → 传统商业数据库，可能面临信创改造压力
- DM/Kingbase → 信创改造进行中
- MySQL/PostgreSQL → 开源数据库，迁移成本低

#### 3. 中间件/框架

**识别关键词**：
- **消息队列**：Kafka, RabbitMQ, RocketMQ
- **缓存**：Redis, Memcached
- **搜索引擎**：Elasticsearch, Solr
- **服务框架**：Dubbo, Spring Cloud, K8s

**技术特征推断**：
- Kafka + Elasticsearch → 大数据架构
- Dubbo/Spring Cloud → 微服务架构
- K8s + Docker → 容器化/云原生

#### 4. 认证/集成

**识别关键词**：
- **AD/LDAP**：Active Directory, LDAP, SSO
- **国产认证**：PKI, CA, 数字证书
- **OAuth**：OAuth2.0, SAML

**技术特征推断**：
- AD → 微软技术栈，Windows 环境
- LDAP → 企业级认证，可能需要集成
- PKI/CA → 金融/政府行业，安全要求高

#### 5. 部署/运维

**识别关键词**：
- **虚拟化**：VMware, OpenStack
- **容器化**：Docker, Kubernetes, K8s
- **云平台**：阿里云, 腾讯云, 华为云, 私有云
- **信创云**：麒麟云, 统信云, 华为云Stack

**技术特征推断**：
- VMware → 传统虚拟化，可能面临信创改造
- K8s → 云原生，容器化部署
- 私有云/信创云 → 国企/政府，数据安全要求高

### 第三步：技术债务信号识别

从 JD 中识别技术债务信号词：

#### 1. 重构信号

**信号词库**：
- 重构、遗留系统、老旧、替换、升级
- 优化、改造、迁移、现代化

**示例 JD 文本**：
- "负责核心系统的重构和优化"
- "遗留系统的改造和维护"

**技术债务推测**：
- 现有系统老旧，维护成本高
- 存在技术升级需求
- 可能是 WPS 365 的切入机会

#### 2. 性能问题信号

**信号词库**：
- 高并发、延迟、卡顿、慢查询、优化
- 性能瓶颈、扩容、压测

**示例 JD 文本**：
- "负责系统性能优化，解决高并发问题"
- "慢查询优化，数据库调优"

**技术债务推测**：
- 现有系统性能瓶颈
- 可能需要架构升级
- 协同办公系统可能也存在性能问题

#### 3. 集成痛点信号

**信号词库**：
- 数据孤岛、打通、对接、集成
- 接口开发、系统对接、数据同步

**示例 JD 文本**：
- "负责多个系统的对接和集成"
- "打破数据孤岛，实现系统互联互通"

**技术债务推测**：
- 系统分散，数据孤岛严重
- 协同办公需求强烈
- WPS 365 的一体化平台有优势

#### 4. 信创改造信号

**信号词库**：
- 信创、国产化、自主可控、迁移
- 达梦、人大金仓、麒麟、统信

**示例 JD 文本**：
- "负责信创改造项目，迁移到国产化平台"
- "熟悉达梦数据库、麒麟操作系统"

**技术债务推测**：
- 正在进行信创改造
- 现有国外产品需要替换
- WPS 365 信创版是机会

### 第四步：WPS 365 部署兼容性分析

基于技术栈分析 WPS 365 部署兼容性：

#### 兼容性分析表

| 技术栈特征 | 推断环境 | WPS 365 推荐方案 | 实施难度 | 兼容性评分 |
|-----------|---------|-----------------|----------|------------|
| .NET + AD | 传统微软 | Windows 客户端 + AD集成 | 低 | ⭐⭐⭐⭐⭐ |
| Java + LDAP | 企业级 Java | Web端 + LDAP集成 | 中 | ⭐⭐⭐⭐ |
| K8s + 微服务 | 云原生 | 容器化部署 | 高 | ⭐⭐⭐ |
| 信创国产化 | 国产化优先 | 国产环境全套 | 中 | ⭐⭐⭐⭐⭐ |
| Oracle + DB2 | 传统商业DB | 信创版 + 数据迁移支持 | 高 | ⭐⭐⭐ |
| MySQL + Redis | 开源技术栈 | 标准版 + API集成 | 低 | ⭐⭐⭐⭐⭐ |

**实施难度说明**：
- **低**：标准化部署，无需定制
- **中**：需要一定集成和配置
- **高**：需要大量定制开发或架构调整

**兼容性评分说明**：
- ⭐⭐⭐⭐⭐：完美兼容，开箱即用
- ⭐⭐⭐⭐：兼容良好，少量配置
- ⭐⭐⭐：兼容可用，需要集成
- ⭐⭐：兼容有限，需要定制
