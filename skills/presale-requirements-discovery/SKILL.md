# WPS 365 需求挖掘助手技能

基于 JTBD（Jobs-to-be-Done）分析法和 SPIN 提问技术，挖掘客户隐性痛点，生成顾问式诊断脚本，将普通拜访转化为"顾问式诊断"。

## 核心价值

- **挖掘隐性痛点**：从 JTBD 视角分析日常工作中的"摩擦点"
- **SPIN 提问脚本**：S/P/I/N 四类问题设计，不让客户反感
- **后果牵连**：引导客户自己说出不解决问题的后果
- **价值畅想**：让客户畅想解决问题后的美好状态
- **模拟实战**：角色扮演，模拟"冷淡型"客户回应

## 触发条件

当用户提供以下任一信息时，启动本技能：

1. **需要挖掘客户痛点**："帮我挖掘某公司的隐性需求"
2. **生成提问脚本**："为 CEO 拜访生成 SPIN 提问脚本"
3. **顾问式诊断准备**："我要拜访某客户，如何提问效果好？"
4. **需求深度挖掘**："如何挖掘客户未明确表达的需求？"

**示例**：
- "帮我挖掘某轨道交通公司运管中心的隐性需求"
- "为应急指挥场景生成 SPIN 提问脚本"
- "我要拜访某大型国企 CIO，需要顾问式诊断脚本"

## 使用流程

### 第一步：理解客户场景

#### 信息收集

从用户输入中提取关键信息：
- **target**：客户公司名称/行业
- **role**：目标角色
- **scenario**：具体业务场景
- **current_status**：客户现状（可选）
- **job_focus**：核心任务关注点（可选）

#### 角色关注点分析

基于 `decision-chain-framework.md` 分析目标角色的核心关注点：

| 角色 | 核心关注 | 首选策略 |
|------|----------|----------|
| CEO/总经理 | 战略价值、ROI、风险控制、竞争优势 | 影响力 + 业务增长 |
| CIO/CTO | 技术架构、运维成本、安全性、可扩展性 | 提效 + 安全 |
| 业务负责人 | 业务效率、协作体验、业务价值 | 提效 + 业务增长 |
| IT 经理 | 技术实现、兼容性、权限管理、数据迁移 | 技术细节 + 安全 |

---

### 第二步：JTBD 痛点挖掘分析

#### Job（核心任务）分析

**任务定义**：客户在场景中要完成的核心任务是什么？

**分析维度**：
- **任务描述**：具体要做什么？
- **任务场景**：在什么情况下做？
- **成功标准**：怎么才算做好？

**示例**：
```
场景：多地协同的应急指挥
Job：在紧急情况下，快速协调多地资源，形成统一指挥方案
成功标准：5分钟内完成信息同步，15分钟内形成初步方案
```

---

#### Friction Points（摩擦点）识别

**摩擦点分类**：

| 类别 | 显性痛点（客户能感知） | 隐性痛点（客户未察觉） | 挖掘方法 |
|------|---------------------|---------------------|---------|
| 信息获取 | 信息散落各处，查找困难 | 信息过载，关键信息被淹没 | 假设性问题推演 |
| 流程协作 | 跨部门沟通成本高 | 默契缺失，反复确认 | 行为观察 + 时间分析 |
| 工具支持 | 系统多，切换频繁 | 工具碎片化导致思维中断 | 工作流分析 |
| 决策支持 | 数据获取慢 | 数据不完整，决策信心不足 | 决策场景还原 |

---

#### Latent Needs（隐性需求）挖掘

**隐性需求特征**：
- 用户未明确表达的需求
- 用户自己未察觉的问题
- 创新机会点

**挖掘技巧**：
- **场景化假设**："如果遇到XX情况，会怎样？"
- **极端情况推演**："最坏的情况下是什么？"
- **理想状态引导**："如果能完美解决，希望是什么样子？"

---

### 第三步：生成 SPIN 提问脚本

#### S - Situation（背景确认）

**目标**：建立安全氛围，收集基础信息

**原则**：
- 不冒犯、让客户舒适
- 基于公开信息的确认
- 引导式非封闭式问题

**数量**：2个问题，总时长1-2分钟

**设计模板**：
```
问题1：基于公开信息的确认
"张总，了解到贵公司在XX方面做了很多投入，目前进展如何？"
- 设计意图：确认现状，建立共鸣
- 预期回应：正面描述 + 过渡到挑战

问题2：引导式业务场景
"在XX场景中，团队是如何协作的？"
- 设计意图：了解工作流程，为后续铺垫
- 预期回应：描述流程（可能暴露痛点）
```

---

#### P - Problem（问题诱导）

**目标**：钩出客户感受到的痛点

**原则**：
- 循序渐进、由表及里
- 从行业共性到客户个性
- 假设性问题引导

**数量**：3个问题，总时长2-3分钟

**设计模板**：
```
问题1：行业共性问题
"在XX行业中，很多客户反馈存在XX问题，贵公司是否也有类似情况？"
- 针对痛点：JTBD摩擦点-显性痛点
- 设计意图：降低防御，"不是只有你有这个问题"

问题2：场景化痛点
"在XX场景中，团队是否遇到过XX情况？"
- 针对痛点：JTBD摩擦点-流程协作
- 设计意图：具体化，让客户有代入感

问题3：隐性痛点诱导
"如果遇到XX情况，目前是怎么处理的？"
- 针对痛点：JTBD摩擦点-隐性痛点
- 设计意图：挖掘未察觉的痛点
```

---

#### I - Implication（后果牵连）

**目标**：让客户自己说出后果

**原则**：
- 客户自述 > 售前人员说
- 假设性问题推演
- 关联 KPI/考核

**数量**：2个问题，总时长1-2分钟

**设计模板**：
```
问题1：短期后果推演
"如果XX问题持续存在，对XX指标会有多大影响？"
- 关联KPI：效率指标/成本指标
- 设计意图：让客户量化损失

问题2：长期后果关联
"长期来看，这对XX（战略目标/政绩考核/市场竞争）会有什么影响？"
- 关联KPI：战略指标/考核指标
- 设计意图：上升到决策者层面
```

---

#### N - Need-payoff（价值畅想）

**目标**：激发解决问题的渴望

**原则**：
- 具体化、可视化
- 量化价值
- 时间紧迫感营造

**数量**：2个问题，总时长1-2分钟

**设计模板**：
```
问题1：理想状态畅想
"如果XX问题能够解决，您希望达到什么效果？"
- 价值点：让客户自己描述理想状态
- 设计意图：激发需求

问题2：价值量化
"如果能实现XX效果，对XX指标（KPI/考核）会有多大帮助？"
- 价值点：量化价值
- 设计意图：为后续方案铺垫价值证明
```

---

### 第四步：WPS 365 解决方案匹配

#### 痛点-解决方案映射

基于 `wps-365-capabilities.md` 和 `industry-pain-points.md`，建立映射关系：

| 挖掘的痛点 | WPS 365 解决方案 | 预期价值 | 优先级 |
|----------|------------------|----------|--------|
| [痛点1] | [方案] | [量化价值] | 高/中/低 |

**价值证明来源**：
- 行业数据：头部企业平均提升 30%-50%
- 标杆案例：某具体客户的效果
- ROI 分析：投资回报周期

### 第五步：模拟实战演练（可选）

当 `include_simulation=true` 时，生成角色扮演对话。

#### 对话结构

```
场景设定：[模拟场景描述]

对话模拟：
售前：[SPIN问题]
客户：[冷淡回应]

客户：[异议]
售前：[异议应对]

- [建议1]
- [建议2]
```

**回应类型**：
- 冷淡型："还好吧"、"一般"、"没啥问题"
- 质疑型："你们能解决？"、"有什么不一样？"
- 防御型："我们有XX系统"、"已经够用了"

**应对技巧**：
- **冷淡型**：换角度提问，用行业案例激发
- **质疑型**：用案例证明，提供试用机会
- **防御型**：承认现有系统价值，补充能力短板

## 示例

1. 分析角色：运管中心主任 → 关注点（提效 + 安全 + 业务增长）

5. 生成模拟实战演练

## 🎯 核心洞察（JTBD 视角）

### Job（核心任务）
- **任务描述**：在紧急情况下，快速协调多地资源，形成统一指挥方案
- **任务场景**：突发事件发生时，需要多地应急中心协同响应
- **成功标准**：5分钟内完成信息同步，15分钟内形成初步方案

### Friction Points（摩擦点分析）

| 摩擦点类别 | 显性痛点 | 隐性痛点（重点） | 严重度 |
|-----------|---------|-----------------|--------|
| 信息获取 | 信息散落在各地，汇总慢 | 关键信息被淹没，决策依据不完整 | 高 |
| 流程协作 | 需要电话逐一确认 | 信息不同步导致决策反复 | 高 |
| 工具支持 | 预案文档版本混乱 | 预案与实际场景脱节，检索困难 | 中 |

### Latent Needs（隐性需求）
- 需求1：AI 辅助决策，基于历史数据推荐预案
- 需求2：实时信息同步，让所有决策者看到同一画面

## 📋 SPIN 提问脚本

### S - 背景确认问题

**问题1**："张主任，了解到贵公司在应急指挥系统上做了很多投入，目前应急响应的效率如何？"
- **设计意图**：确认现状，建立共鸣
- **预期回应**：正面描述系统功能 + 过渡到实际挑战

**问题2**："在多地协同的场景中，信息是如何在各个中心之间同步的？"
- **设计意图**：了解工作流程，为后续铺垫
- **预期回应**：描述流程（可能暴露沟通成本高的问题）

### P - 问题诱导问题

**问题1**："在应急指挥场景中，很多客户反馈信息汇总慢、决策依赖人工经验，贵中心是否也有类似情况？"
- **针对痛点**：信息获取效率
- **设计意图**：行业共性，降低防御

**问题2**："在紧急情况下，如果需要协调多个外地中心，通常需要多长时间完成信息同步？"
- **针对痛点**：流程协作效率
- **设计意图**：让客户自己说出时间成本

**问题3**："预案文档比较多，在紧急情况下如何快速找到对应的预案？"
- **针对痛点**：工具支持效率
- **设计意图**：挖掘预案管理痛点
