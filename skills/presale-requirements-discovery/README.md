# WPS 365 需求挖掘助手 (Requirements Discovery Assistant)

将普通拜访转化为"顾问式诊断"的 WPS 365 售前技能。基于 JTBD 分析法和 SPIN 提问技术，挖掘客户隐性痛点，生成结构化提问脚本。

---

## 核心价值

| 价值 | 说明 |
|------|------|
| **挖掘隐性痛点** | 从 JTBD 视角分析日常工作中的"摩擦点"，发现客户自己未察觉的问题 |
| **SPIN 提问脚本** | S/P/I/N 四类问题设计，循序渐进引导客户，不让客户反感 |
| **后果牵连** | 引导客户自己说出不解决问题的后果，制造紧迫感 |
| **价值畅想** | 让客户畅想解决问题后的美好状态，激发需求 |
| **模拟实战** | 角色扮演模式，模拟"冷淡型"客户回应，提供应对技巧 |

---

## 方法论

### JTBD 分析法

**Jobs-to-be-Done** 是一种以"任务"为中心的需求分析方法，关注用户想要完成的工作，而非用户本身。

**三要素**：
1. **Job（核心任务）**：客户想要完成什么？
2. **Friction Points（摩擦点）**：完成任务的阻碍是什么？
3. **Latent Needs（隐性需求）**：客户未察觉的需求是什么？

---

### SPIN 提问技术

**SPIN** 是一种经典的顾问式销售提问方法，通过四类问题循序渐进引导客户。

| ���题类型 | 目标 | 数量 | 时长 |
|---------|------|------|------|
| **S - Situation** | 背景确认，建立安全氛围 | 2个 | 1-2分钟 |
| **P - Problem** | 问题诱导，钩出痛点 | 3个 | 2-3分钟 |
| **I - Implication** | 后果牵连，让客户自己说后果 | 2个 | 1-2分钟 |
| **N - Need-payoff** | 价值畅想，激发需求 | 2个 | 1-2分钟 |

**总时长**：约 6-10 分钟

## 快速开始

### 基本用法

```
用户：帮我挖掘某轨道交通公司运管中心的隐性需求，场景是多地协同的应急指挥

系统执行：
1. 分析目标角色：运管中心主任
2. JTBD 分析：识别核心任务和摩擦点
3. 生成 SPIN 提问脚本（S-2、P-3、I-2、N-2）
4. 匹配 WPS 365 解决方案
5. 生成模拟实战演练
```

### 参数说明

| 参数 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| target | string | 是 | 客户公司名称/行业 | "某轨道交通公司" |
| role | string | 是 | 目标角色 | "运管中心主任" |
| scenario | string | 是 | 具体业务场景 | "多地协同的应急指挥" |
| current_status | string | 否 | 客户现状 | "使用传统办公软件" |
| job_focus | string | 否 | 核心任务关注点 | "应急响应速度" |
| question_style | string | 否 | 提问风格 | "consultative"（默认） |
| include_simulation | boolean | 否 | 是否包含模拟演练 | true |

## 输出示例

### JTBD 核心洞察

```markdown
### Job（核心任务）
- **任务描述**：在紧急情况下，快速协调多地资源，形成统一指挥方案
- **任务场景**：突发事件发生时，需要多地应急中心协同响应
- **成功标准**：5分钟内完成信息同步，15分钟内形成初步方案

### Friction Points（摩擦点分析）

| 摩擦点类别 | 显性痛点 | 隐性痛点（重点） | 严重度 |
|-----------|---------|-----------------|--------|
| 信息获取 | 信息散落各地，汇总慢 | 关键信息被淹没，决策依据不完整 | 高 |
| 流程协作 | 需要电话逐一确认 | 信息不同步导致决策反复 | 高 |
| 工具支持 | 预案文档版本混乱 | 预案与实际场景脱节，检索困难 | 中 |
```

### SPIN 提问脚本

```markdown
### S - 背景确认问题

**问题1**："张主任，了解到贵公司在应急指挥系统上做了很多投入，目前应急响应的效率如何？"
- **设计意图**：确认现状，建立共鸣
- **预期回应**：正面描述 + 过渡到挑战

### P - 问题诱导问题

**问题1**："在应急指挥场景中，很多客户反馈信息汇总慢、决策依赖人工经验，贵中心是否也有类似情况？"
- **针对痛点**：信息获取效率
- **设计意图**：行业共性，降低防御

### I - 后果牵连问题

**问题1**："如果信息同步需要更长的时间，对应急响应速度会有多大影响？"
- **关联KPI**：应急响应时效
- **设计意图**：让客户量化损失

### N - 价值畅想问题

**问题1**："如果能将信息同步时间缩短到分钟级，您希望达到什么效果？"
- **价值点**：让客户自己描述理想状态
- **设计意图**：激发需求
```

### 模拟实战演练

```markdown
### 对话模拟
**售前**："张主任，了解到贵公司在应急指挥系统上做了很多投入，目前应急响应的效率如何？"

**客户**："还行吧，我们有专业系统。"

**售前**："那在多地协同的场景中，信息是如何在各个中心之间同步的？"

**客户**："电话、邮件，都还挺好用的。"

**售前**："了解。很多应急指挥中心反馈，在紧急情况下，如果需要协调多个外地中心，电话逐一确认可能需要15-20分钟。贵中心这边的信息同步流程是怎样的？"

**客户**："差不多吧，不过我们有预案，按流程走就行。"

**售前**："预案确实重要。不过预案文档比较多，在紧急情况下，如果需要在5分钟内找到对应预案并分发到各个中心，目前的流程能支持吗？"

**客户**："这个...确实有时候找得慢一点。"

**技巧建议**：
- 冷淡回应时，用"很多客户"案例降低防御
- 用具体数据（15-20分钟、5分钟）让客户有代入感
- 强调"补充能力"而非"替换系统"，降低决策压力
```

## 质量检查清单

使用本技能生成的分析报告后，请进行以下质量检查：

- [ ] **JTBD 分析完整**：Job、Friction Points、Latent Needs 三个部分都有实质性内容
- [ ] **摩擦点识别准确**：区分显性痛点和隐性痛点
- [ ] **SPIN 问题设计合理**：S-2问题、P-3问题、I-2问题、N-2问题
- [ ] **问题设计意图清晰**：每个问题都有明确的设计意图
- [ ] **后果关联 KPI**：Implication 问题关联具体 KPI
- [ ] **价值量化具体**：Need-payoff 问题有量化预期
- [ ] **WPS 365 匹配准确**：基于框架文件的准确匹配
- [ ] **角色关注点匹配**：针对目标角色的核心诉求
- [ ] **模拟对话真实**：回应符合"冷淡型"客户特征
